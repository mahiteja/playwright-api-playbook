Playwright API Testing Playbook

Practical, field-tested guide for API testing with Playwright. It focuses on request lifecycle, common HTTP patterns, auth flows teams actually implement, mocking strategies, CI/CD setup, and realistic tradeoffs versus Postman/Newman, RestAssured, Supertest, Karate, Cypress, and k6.

Core API Testing Capabilities

Request context lifecycle, HTTP method patterns, payload handling, response assertions, and request chaining you will use in real suites.

1.1 request Context Setup and Teardown

APIRequestContext is the central object for API calls. It holds connection pooling, base URL, default headers, TLS settings, and cookie storage. Create it via request.newContext() (module-level singleton for the suite) or use the built-in request fixture (one per test). In practice, dispose module-level contexts in afterAll to avoid leaked sockets between runs.

  • Module import: import { request } from '@playwright/test' gives the factory. The request test fixture (destructured from the test callback) is auto-provisioned per test.
  • baseURL: Prepended to relative paths. Set it once so request code stays readable.
  • extraHTTPHeaders: Merged into outgoing requests by default. Per-request headers override these.
  • ignoreHTTPSErrors: Set true only in controlled lab environments against self-signed certs.
TypeScript — tests/api-context.spec.ts 
import { test, expect, request, APIRequestContext } from '@playwright/test';

// Shared context for the entire spec file.
// Eliminates per-test TLS negotiation and connection overhead.
let api: APIRequestContext;

test.beforeAll(async () => {
  api = await request.newContext({
    baseURL: 'https://jsonplaceholder.typicode.com',
    extraHTTPHeaders: {
      'Accept':       'application/json',
      'Content-Type': 'application/json',
      'X-Client':     'playwright-suite',
    },
    ignoreHTTPSErrors: false,
    timeout: 30_000,
  });
});

test.afterAll(async () => {
  await api.dispose(); // releases connection pool + storage
});

test('context is functional', async () => {
  const res = await api.get('/posts/1');
  expect(res.ok()).toBeTruthy();
  expect(res.status()).toBe(200);
});

// --- OR: use the built-in request fixture (no setup needed) ---

test('built-in request fixture', async ({ request }) => {
  // Scoped to this test; baseURL comes from playwright.config use.baseURL
  const res = await request.get('/posts/1');
  expect(res.status()).toBe(200);
});

Use the built-in request fixture when each test needs true session isolation (fresh cookies, independent auth tokens) or when you want zero setup boilerplate for a quick smoke test.

1.2 Making GET, POST, PUT, PATCH, DELETE Requests

Playwright gives each HTTP verb a direct method on APIRequestContext, and in day-to-day test code they behave consistently. You can usually keep one mental model: send payload via data/form/multipart, add headers/params as needed, and control strictness with timeout and failOnStatusCode.

TypeScript — tests/http-methods.spec.ts 
import { test, expect } from '@playwright/test';

const BASE = 'https://jsonplaceholder.typicode.com';

test('GET — retrieve a resource', async ({ request }) => {
  const res = await request.get(`${BASE}/posts/1`);
  expect(res.status()).toBe(200);
  const body = await res.json();
  expect(body).toHaveProperty('id', 1);
  expect(body).toHaveProperty('userId');
  expect(typeof body.title).toBe('string');
});

test('POST — create a resource', async ({ request }) => {
  const res = await request.post(`${BASE}/posts`, {
    data: {
      title: 'Playwright API Test',
      body:  'Automated POST',
      userId: 1,
    },
  });
  expect(res.status()).toBe(201);
  const created = await res.json();
  expect(created).toHaveProperty('id');
  expect(created.title).toBe('Playwright API Test');
});

test('PUT — replace a resource', async ({ request }) => {
  const res = await request.put(`${BASE}/posts/1`, {
    data: {
      id:     1,
      title:  'Fully Replaced',
      body:   'New content',
      userId: 1,
    },
  });
  expect(res.status()).toBe(200);
  const body = await res.json();
  expect(body.title).toBe('Fully Replaced');
});

test('PATCH — partial update', async ({ request }) => {
  const res = await request.patch(`${BASE}/posts/1`, {
    data: { title: 'Patched Title Only' },
  });
  expect(res.status()).toBe(200);
  const body = await res.json();
  expect(body.title).toBe('Patched Title Only');
  // original userId and body fields remain
});

test('DELETE — remove a resource', async ({ request }) => {
  const res = await request.delete(`${BASE}/posts/1`);
  // JSONPlaceholder returns 200 with empty body for DELETE
  expect(res.status()).toBe(200);
});

test('HEAD — check resource existence', async ({ request }) => {
  const res = await request.head(`${BASE}/posts/1`);
  expect(res.status()).toBe(200);
  // No body in HEAD responses
  const text = await res.text();
  expect(text).toBe('');
});

Where each verb is useful GET for read-and-assert contracts. POST for data seeding before UI tests (non-idempotent — retrying creates duplicates). PUT/PATCH for update regression tests. DELETE in finally blocks for test cleanup. HEAD for lightweight liveness probes — returns status and headers with zero body transfer. Use OPTIONS to validate CORS preflight policy on protected cross-origin endpoints.
Practical note — HEAD for fast health checks Prefer request.head('/health') over request.get('/health') in smoke tests that only check HTTP status. HEAD never transfers a body, making it ideal for high-frequency liveness probes or as a pre-check before downloading large payloads.

1.3 Request Headers, Query Params, and Body Payloads

Playwright supports four body encoding strategies controlled by the option key you choose: data (JSON or raw string), form (URL-encoded), multipart (multipart/form-data), or raw data with an explicit Content-Type header. Query parameters are passed as an object to params — no manual URL construction needed.

TypeScript — tests/request-config.spec.ts 
import { test, expect } from '@playwright/test';

// ── Custom headers ──────────────────────────────────────────
test('custom per-request headers', async ({ request }) => {
  const res = await request.get('https://httpbin.org/headers', {
    headers: {
      'X-Trace-ID':   crypto.randomUUID(),
      'X-Source':     'playwright',
      'Accept-Language': 'en-US,en;q=0.9',
    },
  });
  const body = await res.json();
  expect(body.headers['X-Source']).toBe('playwright');
});

// ── Query parameters ────────────────────────────────────────
test('query parameters as object', async ({ request }) => {
  const res = await request.get('https://httpbin.org/get', {
    params: {
      page:   1,
      limit:  25,
      status: 'active',
      tags:   ['api', 'playwright'],  // serialised as repeated params
    },
  });
  expect(res.status()).toBe(200);
  const body = await res.json();
  expect(body.args.page).toBe('1');
  expect(body.args.limit).toBe('25');
});

// ── JSON body ───────────────────────────────────────────────
test('JSON body (data:)', async ({ request }) => {
  const res = await request.post('https://httpbin.org/post', {
    data: {
      username: 'alice',
      roles:    ['admin', 'editor'],
      config:   { theme: 'dark', notifications: true },
    },
  });
  const body = await res.json();
  expect(body.json.username).toBe('alice');
  expect(body.json.roles).toContain('admin');
});

// ── URL-encoded form ────────────────────────────────────────
test('form-urlencoded body (form:)', async ({ request }) => {
  const res = await request.post('https://httpbin.org/post', {
    form: {
      grant_type:    'password',
      username:      'alice',
      password:      'secret',
      client_id:     'myapp',
    },
  });
  const body = await res.json();
  expect(body.form.grant_type).toBe('password');
});

// ── Multipart file upload ────────────────────────────────────
test('multipart/form-data upload (multipart:)', async ({ request }) => {
  const csvContent = Buffer.from('id,name\n1,alice\n2,bob');
  const res = await request.post('https://httpbin.org/post', {
    multipart: {
      report: {
        name:     'report.csv',
        mimeType: 'text/csv',
        buffer:   csvContent,
      },
      source:   'playwright',
      version:  '2',
    },
  });
  expect(res.status()).toBe(200);
  const body = await res.json();
  expect(body.files).toHaveProperty('report');
  expect(body.form.source).toBe('playwright');
});

// ── Raw XML / custom Content-Type ───────────────────────────
test('raw XML body', async ({ request }) => {
  const xml = '<?xml version="1.0"?><order><item>widget</item><qty>5</qty></order>';
  const res = await request.post('https://httpbin.org/post', {
    data:    xml,
    headers: { 'Content-Type': 'application/xml' },
  });
  expect(res.status()).toBe(200);
});
Encoding decision tree Use data: object for JSON (sets Content-Type: application/json automatically). Use form: for OAuth token endpoints that require application/x-www-form-urlencoded. Use multipart: for file uploads. Use data: string with an explicit Content-Type header for XML, NDJSON, or other formats.
Practical note — header precedence Per-request headers are deep-merged with extraHTTPHeaders from the context, with the per-request value winning on conflicts. Use this to override a global Authorization header for specific negative-path tests without creating an entirely new context.

1.4 Response Assertions

APIResponse exposes status(), ok(), headers(), json(), text(), body(), and url(). Combine these with Playwright's expect matchers—including asymmetric matchers like expect.any(), expect.stringMatching(), and expect.objectContaining()—for precise structural assertions without depending on volatile fields.

TypeScript — tests/assertions.spec.ts 
import { test, expect } from '@playwright/test';

test('status code and ok flag', async ({ request }) => {
  const res = await request.get('https://jsonplaceholder.typicode.com/posts/1');

  expect(res.status()).toBe(200);
  expect(res.statusText()).toBe('OK');
  expect(res.ok()).toBeTruthy();   // true for 200-299

  // Negative assertions
  const missing = await request.get('https://jsonplaceholder.typicode.com/posts/99999');
  expect(missing.status()).toBe(404);
  expect(missing.ok()).toBeFalsy();
});

test('response headers', async ({ request }) => {
  const res = await request.get('https://jsonplaceholder.typicode.com/posts/1');

  // All headers as a flat object (lower-cased keys)
  const headers = res.headers();
  expect(headers['content-type']).toContain('application/json');

  // Single header (null-safe)
  const ct = res.headerValue('content-type');
  expect(ct).not.toBeNull();

  // Header present check
  expect(res.headersArray().map(h => h.name.toLowerCase())).toContain('content-type');
});

test('JSON body — exact and partial matching', async ({ request }) => {
  const res = await request.get('https://jsonplaceholder.typicode.com/posts/1');
  const body = await res.json();

  // Exact match
  expect(body.id).toBe(1);
  expect(body.userId).toBe(1);

  // Partial match — ignores additional fields
  expect(body).toMatchObject({
    id:     1,
    title:  expect.any(String),
    body:   expect.any(String),
    userId: expect.any(Number),
  });
});

test('array response assertions', async ({ request }) => {
  const res = await request.get('https://jsonplaceholder.typicode.com/posts');
  const posts = await res.json();

  expect(Array.isArray(posts)).toBeTruthy();
  expect(posts.length).toBeGreaterThan(0);
  expect(posts).toHaveLength(100);

  // Every item matches shape
  for (const post of posts) {
    expect(post).toMatchObject({
      id:     expect.any(Number),
      title:  expect.any(String),
      userId: expect.any(Number),
    });
  }
});

test('nested object and regex assertions', async ({ request }) => {
  const res = await request.get('https://jsonplaceholder.typicode.com/users/1');
  const user = await res.json();

  expect(user).toEqual(
    expect.objectContaining({
      id:    expect.any(Number),
      name:  expect.any(String),
      email: expect.stringMatching(/@/),
      address: expect.objectContaining({
        street:  expect.any(String),
        city:    expect.any(String),
        zipcode: expect.stringMatching(/\d/),
        geo: expect.objectContaining({
          lat: expect.any(String),
          lng: expect.any(String),
        }),
      }),
    })
  );
});

test('binary response body', async ({ request }) => {
  const res = await request.get('https://httpbin.org/image/png');
  const buffer = await res.body();

  expect(buffer).toBeInstanceOf(Buffer);
  expect(buffer.length).toBeGreaterThan(0);
  // PNG magic bytes: 89 50 4E 47
  expect(buffer[0]).toBe(0x89);
  expect(buffer[1]).toBe(0x50); // 'P'
  expect(buffer[2]).toBe(0x4E); // 'N'
  expect(buffer[3]).toBe(0x47); // 'G'
});
Where this is most useful Strong assertions are what keep API suites trustworthy over time. In practice, partial matching with toMatchObject() and expect.objectContaining() survives schema growth better than exact object equality. A good pattern is to guard quickly with res.ok(), then assert the business-critical fields deeply.
Practical note — never assert volatile fields with toBe() Avoid asserting createdAt, updatedAt, UUIDs, ETags, or request IDs directly with toBe(). Assert their type and format instead: expect(body.createdAt).toMatch(/\d{4}-\d{2}-\d{2}/). This prevents false failures when test data is regenerated in CI or when server clocks drift.

1.5 Response Body Extraction and Request Chaining

Chain requests by extracting fields from one response and injecting them into the next. This pattern drives full CRUD flows, covers dependency graphs (e.g., create order → add items → checkout), and seeds test data while validating the resulting state.

TypeScript — tests/chaining.spec.ts 
import { test, expect } from '@playwright/test';

const BASE = 'https://jsonplaceholder.typicode.com';

// ── Full CRUD chain ─────────────────────────────────────────
test('CRUD: create → read → update → delete', async ({ request }) => {
  // CREATE
  const createRes = await request.post(`${BASE}/posts`, {
    data: { title: 'Draft', body: 'Initial content', userId: 1 },
  });
  expect(createRes.status()).toBe(201);
  const { id } = await createRes.json();
  expect(typeof id).toBe('number');

  // READ
  const readRes = await request.get(`${BASE}/posts/${id}`);
  expect(readRes.status()).toBe(200);
  const post = await readRes.json();
  expect(post.id).toBe(id);

  // UPDATE
  const patchRes = await request.patch(`${BASE}/posts/${id}`, {
    data: { title: 'Published' },
  });
  expect(patchRes.status()).toBe(200);
  const updated = await patchRes.json();
  expect(updated.title).toBe('Published');

  // DELETE
  const delRes = await request.delete(`${BASE}/posts/${id}`);
  expect(delRes.status()).toBe(200);
});

// ── Token → authenticated call chain ────────────────────────
test('login → use session token in subsequent calls', async ({ request }) => {
  // Step 1: Obtain access token
  const loginRes = await request.post('https://dummyjson.com/auth/login', {
    data: { username: 'kminchelle', password: '0lelplR' },
  });
  expect(loginRes.status()).toBe(200);

  const { accessToken } = await loginRes.json();
  expect(accessToken).toBeTruthy();

  // Step 2: Use token for a protected endpoint
  const profileRes = await request.get('https://dummyjson.com/auth/me', {
    headers: { Authorization: `Bearer ${accessToken}` },
  });
  expect(profileRes.status()).toBe(200);

  const profile = await profileRes.json();
  expect(profile).toHaveProperty('id');
  expect(profile).toHaveProperty('username', 'kminchelle');

  // Step 3: Use user ID from profile in another request
  const cartRes = await request.get(`https://dummyjson.com/carts/user/${profile.id}`);
  expect(cartRes.status()).toBe(200);
  const cartData = await cartRes.json();
  expect(cartData).toHaveProperty('carts');
});
Where this is most useful Request chaining is essential for workflows where intermediate state cannot be queried independently: OAuth flows (exchange code → get token → call protected endpoint), order systems (create cart → add items → checkout → verify confirmation), and onboarding sequences. It also replaces complex test fixtures by building state incrementally inside a single test.
Practical note — always clean up with finally Wrap chained CRUD tests in try { ... } finally { await request.delete(...) } so DELETE cleanup runs even when an intermediate assertion throws. Without this, failed tests leave orphaned data that accumulates across runs and causes false failures in later tests.

1.6 Global Request Context vs. Page-bound Context

Playwright offers two distinct contexts for API calls. The global context (request.newContext() or the request fixture) is completely independent of any browser — it does not share cookies or storage. The page-bound context (page.request or page.context().request) shares cookies and storage with the live browser page, making it ideal for testing authenticated REST endpoints after a UI login.

TypeScript — tests/context-types.spec.ts 
import { test, expect, request } from '@playwright/test';

// ── Global context: no browser, no shared cookies ───────────
test.describe('Global API context', () => {
  test('fetch resource without browser', async ({ request }) => {
    // `request` fixture: isolated, does not share any browser cookies
    const res = await request.get('https://jsonplaceholder.typicode.com/todos/1');
    expect(res.status()).toBe(200);
    const todo = await res.json();
    expect(todo).toHaveProperty('id', 1);
  });
});

// ── Module-level singleton context ─────────────────────────
import { APIRequestContext } from '@playwright/test';

let globalApi: APIRequestContext;

test.beforeAll(async () => {
  globalApi = await request.newContext({
    baseURL: 'https://jsonplaceholder.typicode.com',
  });
});

test.afterAll(() => globalApi.dispose());

test('module-level context survives across tests', async () => {
  const [posts, users] = await Promise.all([
    globalApi.get('/posts/1'),
    globalApi.get('/users/1'),
  ]);
  expect(posts.status()).toBe(200);
  expect(users.status()).toBe(200);
});

// ── Page-bound context: shares browser session cookies ──────
test('page.request inherits browser session', async ({ page }) => {
  // Login via browser UI
  await page.goto('https://the-internet.herokuapp.com/login');
  await page.fill('#username', 'tomsmith');
  await page.fill('#password', 'SuperSecretPassword!');
  await page.click('button[type="submit"]');
  await page.waitForURL('**/secure');

  // page.request carries the session cookie set during login
  const res = await page.request.get(
    'https://the-internet.herokuapp.com/secure'
  );
  expect(res.status()).toBe(200);
  const text = await res.text();
  expect(text).toContain('You logged into a secure area');
});
Common mistake Using page.request vs. the request fixture interchangeably. They are NOT equivalent: the fixture has no browser session, page.request does. Mix them up and authenticated calls unexpectedly return 401.
Practical note — decision rule for context choice Default to the request fixture for pure API tests — it is lighter (no browser overhead) and fully isolated. Switch to page.request only when the test also drives browser UI and you need cookie continuity between the browser session and API calls (e.g., CSRF token validation, session-authenticated APIs without explicit Authorization headers).

1.7 Intercepting and Mocking API Calls with page.route

page.route() registers a handler that intercepts browser-initiated network requests matching a URL pattern. The handler receives a Route object and can: fulfill with a mock response, abort the request, forward to the real server unmodified (route.continue()), or fetch the real response and modify it before returning to the page. This is distinct from API-context calls — it applies only to requests the browser page itself makes.

TypeScript — tests/mocking.spec.ts 
import { test, expect } from '@playwright/test';

// ── Fulfill with a static mock ──────────────────────────────
test('mock GET /api/users with fixture data', async ({ page }) => {
  const mockUsers = [
    { id: 1, name: 'Alice', role: 'admin' },
    { id: 2, name: 'Bob',   role: 'viewer' },
  ];

  await page.route('**/api/users', async route => {
    await route.fulfill({
      status:      200,
      contentType: 'application/json',
      body:        JSON.stringify(mockUsers),
    });
  });

  await page.goto('/users');
  await expect(page.locator('[data-testid="user-name"]').first()).toHaveText('Alice');
});

// ── Abort a request ─────────────────────────────────────────
test('abort third-party analytics calls', async ({ page }) => {
  await page.route('**/analytics/**', route => route.abort());
  await page.goto('https://example.com');
  // analytics requests are blocked; page loads without them
});

// ── Intercept, modify, forward ──────────────────────────────
test('inject extra field into API response', async ({ page }) => {
  await page.route('**/api/profile', async route => {
    // Fetch the real response
    const realResponse = await route.fetch();
    const body = await realResponse.json();

    // Augment it
    body.featureFlags = { darkMode: true, betaEnabled: false };

    await route.fulfill({
      response:    realResponse,
      body:        JSON.stringify(body),
      contentType: 'application/json',
    });
  });

  await page.goto('/profile');
  // Test behaviour that depends on featureFlags being present
});

// ── Simulate error responses ────────────────────────────────
test('UI handles 500 Internal Server Error gracefully', async ({ page }) => {
  await page.route('**/api/orders', route =>
    route.fulfill({
      status: 500,
      body:   JSON.stringify({ error: 'Database unavailable' }),
      contentType: 'application/json',
    })
  );

  await page.goto('/orders');
  await expect(page.locator('[data-testid="error-banner"]'))
    .toContainText('unavailable');
});

// ── Mock with a HAR file ────────────────────────────────────
test('replay from recorded HAR', async ({ page }) => {
  // Record once:  await page.routeFromHAR('./fixtures/api.har', { update: true });
  await page.routeFromHAR('./fixtures/api.har', {
    notFound: 'fallback', // un-matched requests pass through
  });
  await page.goto('/dashboard');
});

    participant P as Browser Page
    participant R as Route Handler
    participant A as Real API

    Note over P: Handler registered

Where this is most useful page.route() is indispensable for isolating UI tests from unreliable third-party APIs (payment gateways, analytics, address lookups), testing edge cases that are hard to reproduce on a real backend (rate-limit responses, 503 errors, partial payloads), and speeding up test suites by eliminating network round-trips to slow external services.
Practical note — route.fallback() for selective interception Chain multiple page.route() handlers and use route.fallback() to pass requests through to the next handler or the real network. This lets you intercept only /api/payments in a beforeEach while letting all other requests pass through — without a catch-all handler that breaks legitimate calls.

1.8 Combining UI and API Testing in a Single Test

The most distinctive Playwright capability: a single test can create data via API, verify it in the browser UI, and clean up via API — all with consistent state. This eliminates slow UI-only setup flows while keeping full end-to-end coverage for the critical path.

TypeScript — tests/hybrid.spec.ts 
import { test, expect } from '@playwright/test';

// Pattern 1: API seed → UI verify
test('create post via API, verify it appears in UI', async ({ request, page }) => {
  // Fast API setup — no UI clicks for data seeding
  const createRes = await request.post('https://jsonplaceholder.typicode.com/posts', {
    data: { title: 'API-seeded Post', body: 'Seeded via API', userId: 1 },
  });
  expect(createRes.status()).toBe(201);
  const { id } = await createRes.json();

  // Verify presence in the UI
  await page.goto(`https://jsonplaceholder.typicode.com/posts/${id}`);
  // (JSONPlaceholder returns JSON, not HTML, so this is illustrative)
  const bodyText = await page.locator('body').textContent();
  expect(bodyText).toBeTruthy();
});

// Pattern 2: UI action → API state verification
test('UI form submission reflects in API response', async ({ page, request }) => {
  await page.goto('https://demoqa.com/automation-practice-form');

1.9 GraphQL API Testing

GraphQL APIs use a single POST endpoint with a JSON body containing query and optional variables. The critical difference from REST: the server returns HTTP 200 even on application-level errors — you should inspect the errors field alongside data. Use request.post() with a structured data object; Playwright serialises it automatically.

  • Avoid interpolating values into query strings — pass them via the variables key to reduce injection risk and improve server-side query caching.
  • Introspection queries (__schema, __type) are a free contract test — run them in CI to detect breaking schema changes before they reach production.
  • Partial success: GraphQL can return both data and errors simultaneously when some resolvers succeed and others fail.
  • Mock via page.route: intercept **/graphql and branch on requestBody.query to return different fixtures per operation name.

1.10 WebSocket Testing

Playwright intercepts WebSocket connections at the browser level via the page.on('websocket', ...) event. Frames sent and received by the page are observable without modifying the connection. From Playwright v1.48, page.routeWebSocket() enables full mocking: intercept frames from the page, respond with custom payloads, and optionally forward to the real server.

  • Observe mode (page.on('websocket')): inspect all sent/received frames without altering them — ideal for black-box assertions on a real backend.
  • Mock mode (page.routeWebSocket()): intercept and respond without hitting a real server — ideal for CI isolation. Requires Playwright ≥ 1.48.
  • Frame payload: available as event.payload — string for text frames, Buffer for binary.
  • Security assertion: iterate connections array and assert each URL matches an allowed pattern to catch unexpected data egress.
TypeScript — tests/websocket.spec.ts 
import { test, expect } from '@playwright/test';

// ── Observe frames (read-only) ────────────────────────────────

1.11 Server-Sent Events (SSE)

Server-Sent Events use a persistent HTTP connection with Content-Type: text/event-stream. Playwright intercepts them via page.route() — fulfill with a multi-event body string to inject a mock stream, or call request.get() directly to assert the raw SSE format of a live endpoint.

TypeScript — tests/sse.spec.ts 
import { test, expect } from '@playwright/test';

// ── Mock an SSE stream and assert all events received ─────────
test('mock SSE stream — assert all events', async ({ page }) => {
  await page.route('**/api/events', async route => {
    // SSE wire format: each event is "data: \n\n"
    const stream = [
      'data: {"type":"start","jobId":"j-001"}\n\n',
      'data: {"type":"progress","value":33}\n\n',
      'data: {"type":"progress","value":66}\n\n',
      'data: {"type":"complete","result":"success"}\n\n',
    ].join('');

    await route.fulfill({
      status:  200,
      headers: {
        'Content-Type':  'text/event-stream',
        'Cache-Control': 'no-cache',
        'Connection':    'keep-alive',
      },
      body: stream,
    });
  });

  await page.goto('/job-status');

  // Collect events from the page's EventSource
  const events = await page.evaluate(() => {
    return new Promise<object[]>(resolve => {
      const collected: object[] = [];
      const es = new EventSource('/api/events');
      es.onmessage = (e: MessageEvent) => {
        collected.push(JSON.parse(e.data));
        if (collected.length === 4) { es.close(); resolve(collected); }
      };
      setTimeout(() => resolve(collected), 10_000);
    });
  });

  expect(events).toHaveLength(4);
  expect(events[0]).toMatchObject({ type: 'start' });
  expect(events[3]).toMatchObject({ type: 'complete', result: 'success' });
});

// ── Assert live SSE endpoint format ──────────────────────────
test('live SSE endpoint delivers valid event-stream format', async ({ request }) => {
  const res = await request.get('https://api.example.com/stream/status', {
    headers: { Accept: 'text/event-stream' },
    timeout: 5_000,
  });

  expect(res.headers()['content-type']).toContain('text/event-stream');

  const text = await res.text();
  // At least one "data:" line must be present
  expect(text).toMatch(/^data:/m);

  // Parse and assert the first event
  const firstDataLine = text.split('\n').find(l => l.startsWith('data:'));
  const payload = JSON.parse(firstDataLine!.replace('data:', '').trim());
  expect(payload).toHaveProperty('type');
});

2.2 Basic Authentication

HTTP Basic Auth sends Authorization: Basic <base64(user:pass)> on every request. Playwright handles this natively via httpCredentials on the context—no manual Base64 encoding required. Only use over HTTPS; credentials are trivially decodable if intercepted in transit.

2.5 OAuth 2.0 — Authorization Code Flow

The browser-based user-delegated flow. Playwright drives the browser through the authorization redirect, captures the auth code from the callback URL, and exchanges it for tokens. This is the only realistic way to automate this flow end-to-end — other tools require manual token injection.

TypeScript — auth/oauth2-auth-code.spec.ts 
import { test, expect } from '@playwright/test';

const AUTH_URL    = process.env.AUTH_SERVER_URL!;
const CLIENT_ID   = process.env.CLIENT_ID!;
const CLIENT_SEC  = process.env.CLIENT_SECRET!;
const REDIRECT    = 'https://app.example.com/oauth/callback';
const API_URL     = process.env.API_BASE_URL!;

test('OAuth 2.0 authorization code flow', async ({ page }) => {
  // Step 1: Navigate to authorization endpoint
  const authParams = new URLSearchParams({
    response_type: 'code',
    client_id:     CLIENT_ID,
    redirect_uri:  REDIRECT,
    scope:         'openid profile email',
    state:         crypto.randomUUID(), // CSRF protection
  });

  await page.goto(`${AUTH_URL}/authorize?${authParams}`);

  // Step 2: Fill in provider login form
  await page.waitForSelector('#username');
  await page.fill('#username', process.env.TEST_USERNAME!);
  await page.fill('#password', process.env.TEST_PASSWORD!);
  await page.click('button[type="submit"]');

  // Step 3: Handle consent screen (if present)
  if (await page.locator('#consent-accept').isVisible()) {
    await page.click('#consent-accept');
  }

  // Step 4: Capture authorization code from callback URL
  await page.waitForURL(`${REDIRECT}**`);
  const callbackUrl = new URL(page.url());
  const authCode = callbackUrl.searchParams.get('code');
  expect(authCode).toBeTruthy();

  // Step 5: Exchange code for tokens (back-channel, not in browser)
  const tokenRes = await page.request.post(`${AUTH_URL}/token`, {
    form: {
      grant_type:    'authorization_code',
      code:          authCode!,
      redirect_uri:  REDIRECT,
      client_id:     CLIENT_ID,
      client_secret: CLIENT_SEC,
    },
  });

  expect(tokenRes.status()).toBe(200);
  const tokens = await tokenRes.json();
  expect(tokens).toHaveProperty('access_token');
  expect(tokens).toHaveProperty('refresh_token');
  expect(tokens).toHaveProperty('id_token'); // OIDC

  // Step 6: Call protected API with access token
  const apiRes = await page.request.get(`${API_URL}/me`, {
    headers: { Authorization: `Bearer ${tokens.access_token}` },
  });
  expect(apiRes.status()).toBe(200);
  const profile = await apiRes.json();
  expect(profile).toHaveProperty('email');
});

2.7 NTLM / Windows Authentication

Playwright uses the underlying OS HTTP library for NTLM/Negotiate challenge-response. Set httpCredentials with sendImmediately: false — this tells Playwright to wait for the server's 401 challenge before responding, which is the required handshake for NTLM (sending Basic credentials upfront would fail). Domain credentials use the DOMAIN\\user format.

Windows runner requirement NTLM challenge-response relies on the OS Negotiate stack. For full NTLM (not just Basic) to work correctly, run these tests on a Windows CI runner or a Windows Docker image with Active Directory access. On Linux, install libkrb5-dev for Kerberos support.
Practical note — verify the negotiate header value The server sends a WWW-Authenticate: Negotiate or WWW-Authenticate: NTLM challenge header. 2.9 Storing and Reusing Auth State with storageState

storageState serializes cookies, localStorage, and sessionStorage to a JSON file. Perform one login in global-setup.ts, save the state, and all tests load it — eliminating repeated login flows that inflate suite execution time. This is the recommended pattern for any test that requires authentication.

TypeScript — global-setup.ts 
import { request } from '@playwright/test';

// Runs once before the entire test suite
async function globalSetup() {
  // ── Option A: API login (fastest) ──────────────────────────
  const api = await request.newContext({
    baseURL: process.env.APP_URL!,
  });

  const loginRes = await api.post('/api/auth/login', {
    data: {
      username: process.env.TEST_USER!,
      password: process.env.TEST_PASS!,
    },
  });

  if (!loginRes.ok()) {
    throw new Error(`Global setup login failed: ${loginRes.status()}`);
  }

  // Persist cookies (and localStorage if set during login)
  await api.storageState({ path: './auth-state/user.json' });
  await api.dispose();
}

export default globalSetup;

3.1 Fixtures and Reusable API Clients

Playwright's fixture system (test.extend()) is the right abstraction for shared API clients. Each fixture has a well-defined lifecycle (setup → yield → teardown), participates in Playwright's dependency injection, and can be scoped to 'test' or 'worker'. Worker-scoped fixtures create one client per parallel worker rather than one per test.

3.2 Retries and Timeout Configuration

Playwright provides retry logic at three levels: test-level retries (run the entire test again on failure), per-request timeouts, and poll-based assertions (expect.poll) for eventually-consistent endpoints. Combine these to handle flaky external services without masking real bugs.

  timeout: 30_000,

      return state;

Practical note — use intervals that match your system's behaviour For async jobs that typically complete in 5–10 seconds, use intervals: [1000, 2000, 2000, 5000] — polling aggressively early (catching fast completions) then backing off. 3.9 Soft Assertions

expect.soft() records a failure but does not abort the test. All soft assertions are evaluated, then all failures are reported together at the end of the test. This is particularly useful for API response validation: see every broken field in a single run rather than stopping at the first mismatch.

TypeScript — tests/soft-assertions.spec.ts 
import { test, expect } from '@playwright/test';

test('soft assertions on user response fields', async ({ request }) => {
  const res = await request.get('https://jsonplaceholder.typicode.com/users/1');

  // Hard assertion — aborts immediately on failure
  expect(res.status()).toBe(200);

  const user = await res.json();

  // Soft assertions — all evaluated; all failures reported together
  expect.soft(user.id,      'id is a number').toEqual(expect.any(Number));
  expect.soft(user.name,    'name is non-empty').toBeTruthy();
  expect.soft(user.email,   'email contains @').toMatch(/@/);
  expect.soft(user.username,'username is defined').toBeDefined();
  expect.soft(user.phone,   'phone is defined').toBeDefined();
  expect.soft(user.address, 'address has city').toMatchObject({
    city: expect.any(String),
  });
  // Playwright reports ALL soft failures — not just the first
});

// ── Soft-validate every item in a list ───────────────────────
test('validate all users — report all invalid emails', async ({ request }) => {
  const res   = await request.get('https://jsonplaceholder.typicode.com/users');
  const users = await res.json();

  expect(res.status()).toBe(200);

  for (const user of users) {
    expect.soft(
      user.email,
      `user ${user.id}: invalid email format`
    ).toMatch(/^[^\s@]+@[^\s@]+\.[^\s@]+$/);

    expect.soft(
      typeof user.id,
      `user ${user.id}: id must be number type`
    ).toBe('number');
  }
  // Reports ALL failing users, not just the first
});
When to use soft assertions Use expect.soft() when validating multiple independent fields on a single API response, or when iterating a list and wanting a complete failure report. Avoid them for guard assertions where a failure makes subsequent assertions meaningless (e.g., checking status before parsing the body).
Practical note — add a descriptive message to every soft assertion Pass a second string argument: expect.soft(user.email, `user ${user.id} has invalid email`).toMatch(/.../);. Without this, failure messages like “received undefined” give no context about which list item failed. Descriptive messages transform a soft-assertion failure report into an actionable list of exactly which records are broken.

3.10 Response Time and Latency Assertions

Playwright has no dedicated timing API for HTTP requests, but wrapping a request.get() call with performance.now() gives sub-millisecond accuracy. Use this for SLA smoke tests (a single request must complete under a threshold) or aggregate P95 latency checks across multiple samples in the same test.

TypeScript — tests/response-time.spec.ts 
import { test, expect } from '@playwright/test';

const BASE = 'https://jsonplaceholder.typicode.com';

// ── Single endpoint SLA ───────────────────────────────────────
test('GET /posts responds within 500ms', async ({ request }) => {
  const t0  = performance.now();
  const res = await request.get(`${BASE}/posts`);
  const ms  = performance.now() - t0;

  expect(res.status()).toBe(200);
  expect(ms, `Response took ${ms.toFixed(0)}ms — SLA is 500ms`).toBeLessThan(500);
});

// ── Multi-endpoint SLA matrix ─────────────────────────────────
const SLA_MS = 300;
const criticalPaths = ['/posts/1', '/users/1', '/todos/1'];

test.describe(`Critical paths under ${SLA_MS}ms SLA`, () => {
  for (const path of criticalPaths) {
    test(`GET ${path}`, async ({ request }) => {
      const t0  = performance.now();
      const res = await request.get(`${BASE}${path}`);
      const ms  = performance.now() - t0;

      expect(res.status()).toBe(200);
      expect(ms, `${path}: ${ms.toFixed(0)}ms exceeds SLA ${SLA_MS}ms`).toBeLessThan(SLA_MS);
    });
  }
});

// ── P95 latency over N samples ────────────────────────────────
test('P95 latency across 10 samples < 400ms', async ({ request }) => {
  const samples: number[] = [];

  for (let i = 0; i < 10; i++) {
    const t0 = performance.now();
    await request.get(`${BASE}/posts/1`);
    samples.push(performance.now() - t0);
  }

  samples.sort((a, b) => a - b);
  const p95 = samples[Math.ceil(samples.length * 0.95) - 1];
  const avg = samples.reduce((sum, v) => sum + v, 0) / samples.length;

  console.log(`Latency — avg: ${avg.toFixed(1)}ms, p95: ${p95.toFixed(1)}ms`);
  expect(
    p95,
    `P95 (${p95.toFixed(0)}ms) exceeds 400ms threshold`
  ).toBeLessThan(400);
});
Not a substitute for load testing These checks use a single sequential HTTP connection with no concurrency. They verify that an endpoint responds within SLA under zero load — not under realistic traffic. Use k6 or Gatling for load testing alongside these functional latency guards.
Where this is most useful Response-time assertions serve as SLA regression guards — they alert you when a code change or infrastructure change degrades endpoint latency beyond an agreed threshold. They are most valuable in: critical-path smoke tests (POST /checkout must respond under 500ms), database-query regression detection (an added N+1 query doubles response time), and CDN cache-miss detection (cold response 10× slower than warm).

3.11 Snapshot Testing for API Responses

expect(value).toMatchSnapshot(name) writes the value to a __snapshots__ file on first run and diffs against it on subsequent runs. For API responses, this is a low-friction way to catch unintentional contract changes — especially for large nested objects, enum lists, or fixed configuration payloads that are impractical to assert field-by-field.

3.12 Custom API Matchers with expect.extend()

TypeScript — matchers/api.ts 
import { expect, type APIResponse } from '@playwright/test';

expect.extend({

  async toHaveStatus(received: APIResponse, expected: number) {
    const actual = received.status();
    return {
      pass:    actual === expected,
      message: () =>
        `Expected HTTP ${expected}, received HTTP ${actual}\nURL: ${received.url()}`,
    };
  },

  async toBeSuccessfulJSON(received: APIResponse) {
    const status = received.status();
    const ct     = received.headers()['content-type'] ?? '';
    const pass   = received.ok() && ct.includes('application/json');
    return {
      pass,
      message: () =>
        pass
          ? `Expected NOT to be successful JSON`
          : `Expected 2xx + application/json, got ${status} + "${ct}"`,
    };
  },

  async toContainItems(received: APIResponse, minCount: number) {
    const body  = await received.json();
    const arr   = Array.isArray(body)
      ? body
      : (body.data ?? body.items ?? body.results ?? []);
    const count = arr.length;
    return {
      pass:    count >= minCount,
      message: () => `Expected at least ${minCount} items, got ${count}`,
    };
  },

  async toMatchAPIContract(received: APIResponse, schema: { safeParse: Function }) {
    const body   = await received.json();
    const result = schema.safeParse(body);
    return {
      pass:    result.success,
      message: () =>
        result.success
          ? `Expected NOT to match schema`
          : `API contract broken:\n${JSON.stringify(
              result.error.flatten().fieldErrors, null, 2
            )}`,
    };
  },
});

// Extend Playwright's type declarations so TypeScript recognises the matchers
declare module '@playwright/test' {
  interface AsyncMatchers {
    toHaveStatus(expected: number):            Promise<void>;
    toBeSuccessfulJSON():                       Promise<void>;
    toContainItems(minCount: number):          Promise<void>;
    toMatchAPIContract(schema: any):           Promise<void>;
  }
}

3.14 maxRetries on fetch(): ECONNRESET Resilience v1.46

Unstable CI environments — shared runners, NAT gateways, flaky proxy layers — can drop TCP connections mid-request, causing ECONNRESET errors that fail tests for infrastructure reasons rather than actual bugs. The maxRetries option on apiRequestContext.fetch() (and its verb shortcuts) tells Playwright to automatically retry on that specific error class only.

TypeScript — tests/resilient-fetch.spec.ts

// ── Idempotent POST with retry ────────────────────────────────

  // (e.g. upsert by ID). Non-idempotent POST + retry = duplicate records.
  const res = await request.post(`${BASE}/posts`, {

  });

Where this is most useful maxRetries is most valuable in shared CI infrastructure where network connections are frequently reset by overloaded proxies, NAT gateways reaching connection limits, or ephemeral container networking. test('mock adapts behaviour to negotiated subprotocol', async ({ page }) => {

When to Choose Playwright for API Testing

Specific scenarios where Playwright excels and where a different tool is the better fit.

Choose Playwright when…

  • You need UI + API in the same test. Seed data via API, assert in browser, clean up via API — without tool switching or test orchestration glue code.
  • Auth flows span browser and API. OAuth authorization code, SSO redirects, SAML assertions — Playwright can drive the browser through the full flow and extract tokens for API calls in the same test.
  • Your team is already in TypeScript/JavaScript. One language, one runner, one report — no Java or Go expertise required.
  • You need page.route for component isolation. Intercept, mock, or modify API responses during browser tests to isolate frontend behavior from backend state.
  • You want a single CI pipeline for all test types. API, UI, and component tests share the same runner, config, and report format.
  • mTLS or Windows Auth (NTLM) is required. Native support in request.newContext() — no proxy workarounds needed.
  • storageState eliminates repeated logins. One login in global-setup.ts, serialized to JSON, loaded by every test — critical for large suites with authenticated APIs.
  • You need to test multi-tab or multi-context workflows. E.g., validate that an action in Tab A triggers state visible via API that Tab B reflects.

Choose a different tool when…

  • Pure load / performance testing. Use k6 or Gatling. Playwright spawns a browser process per worker — it cannot simulate 500+ concurrent virtual users efficiently.
  • The team is Java-only. Use RestAssured. Forcing a JS toolchain on a Java shop creates maintenance friction without meaningful benefit.
  • Exploratory and manual API testing with rich collaboration features. Use Postman. The Collection Runner, Visualizer, and team workspaces have no Playwright equivalent.
  • Non-engineers write the tests. Use Karate. Its Gherkin DSL is accessible to business analysts and manual QAs who cannot write TypeScript.
  • Testing a Node.js server in-process. Use Supertest. Binding directly to an Express/Fastify app without opening a port is simpler and faster for unit-level route tests.
  • You need JSON Schema generation + API documentation. Use Postman or Stoplight. Playwright has no schema-generation or documentation tooling.
  • Your SOAP/XML contract testing is complex. Use RestAssured or Karate, both with native XPath/GPath assertion support. Playwright treats all bodies as text/JSON.
